> Виды архитектур на уровне приложения: слоистая, модульная, чистая.

## Слоистая архитектура (Layered)

Классическое деление на горизонтальные слои, каждый зависит от слоя ниже:

- **Web / Presentation** (`@Controller`/`@RestController`) — приём HTTP, валидация, DTO.
- **Service / Business** (`@Service`) — бизнес-логика, транзакции.
- **Repository / Persistence** (`@Repository`) — доступ к данным.
- **Domain / Model** — сущности.

Правило: слой обращается только к нижележащему (Web → Service → Repository). Плюсы:
просто, знакомо всем. Минусы: бизнес-логика зависит от инфраструктуры (БД, фреймворк),
слои могут «протекать» (анемичная модель, размазывание логики).

## Модульная архитектура (Modular / по бизнес-фичам)

Код делится не по техническим слоям, а по **бизнес-доменам/модулям** (например,
`orders`, `users`, `payments`). Внутри каждого модуля — свои слои. Модули общаются через
явные интерфейсы/API, внутренние детали инкапсулированы.

Плюсы: высокая связность внутри модуля и слабая связанность между модулями; модуль легко
выделить в отдельный микросервис. В Spring есть **Spring Modulith** для контроля границ
модулей. Минусы: нужна дисциплина в определении границ.

## Чистая архитектура (Clean / гексагональная)

Зависимости направлены **внутрь**, к доменному ядру. В центре — бизнес-правила (Entities,
Use Cases), которые **не знают** о фреймворке, БД и UI. Внешние детали (БД, web, очереди)
— это «адаптеры» на периферии, подключаемые через **порты** (интерфейсы).

```
[ UI / Web ] → [ Use Cases ] → [ Entities ]
[   DB     ] →     (порты определены в ядре, реализованы снаружи)
```

**Правило зависимостей**: внутренние слои ничего не знают о внешних. Доступ к БД описан
интерфейсом (порт) в доменном слое, а реализация (`JpaRepository`) — снаружи и внедряется
через DI.

Плюсы: ядро тестируется без инфраструктуры, легко менять БД/фреймворк, бизнес-логика
изолирована. Минусы: больше абстракций и кода, избыточно для простых CRUD-приложений.
