> AOP. Фреймворк Spring AOP и библиотеки CGLIB & JDK Dynamic proxies. Фреймворк AspectJ и понятие Weaving.

## Spring AOP

Spring AOP реализует AOP **через прокси-объекты** во время выполнения. Когда у бина есть
применимый аспект, контейнер регистрирует не сам бин, а его **прокси**. Все вызовы идут
через прокси, который выполняет advice до/после делегирования настоящему объекту.

Особенности и ограничения Spring AOP:
- Перехватываются **только публичные методы бинов**, вызванные **извне** (через прокси).
- **Self-invocation** (вызов метода того же класса через `this.method()`) AOP **не
  перехватывает** — вызов идёт мимо прокси. Это частый источник багов (например,
  `@Transactional` не срабатывает при внутреннем вызове).
- Join point ограничен вызовом метода (нет перехвата полей, конструкторов).

## JDK Dynamic Proxy

Если у целевого бина есть **интерфейс**, Spring по умолчанию создаёт **JDK-прокси** —
динамический класс, реализующий тот же интерфейс (`java.lang.reflect.Proxy`). Прокси можно
присвоить переменной типа интерфейса, **но не типа конкретного класса**.

## CGLIB Proxy

Если интерфейса нет (или включён `proxyTargetClass=true`), Spring использует **CGLIB** —
генерирует **подкласс** целевого класса в рантайме и переопределяет методы. Поэтому:
- класс и методы **не должны быть `final`** (нельзя унаследовать/переопределить);
- работает и без интерфейсов.

Spring Boot по умолчанию использует CGLIB (`spring.aop.proxy-target-class=true`).

| | JDK Dynamic Proxy | CGLIB |
|---|---|---|
| Условие | есть интерфейс | подкласс класса |
| Механизм | реализует интерфейс | наследует класс |
| Ограничение | бин виден только как интерфейс | нет `final` класса/методов |

## AspectJ

**AspectJ** — полноценный AOP-фреймворк (не ограничен прокси). Поддерживает перехват
полей, конструкторов, self-invocation, работу с не-Spring-объектами. Pointcut-язык AspectJ
Spring переиспользует, но сам механизм у AspectJ другой — **weaving**.

## Weaving (вплетение)

**Weaving** — процесс внедрения аспектного кода в целевые классы. Виды:
- **Runtime weaving** — через прокси в рантайме (это и есть Spring AOP).
- **Compile-time weaving (CTW)** — аспекты вплетаются компилятором AspectJ (`ajc`) при сборке.
- **Post-compile (binary) weaving** — вплетение в уже скомпилированные `.class`/`.jar`.
- **Load-time weaving (LTW)** — вплетение при загрузке классов JVM специальным агентом.

Итог: для большинства задач (логирование, транзакции, безопасность) достаточно
**Spring AOP** (прокси). Полный **AspectJ** нужен, когда требуется перехват за пределами
вызовов публичных методов бинов или нет накладных расходов прокси.
