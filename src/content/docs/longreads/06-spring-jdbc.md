## Введение

Работа с реляционными базами данных — неотъемлемая часть большинства enterprise-приложений на Java. Однако прямое использование низкоуровневых API, таких как JDBC, сопряжено с рядом сложностей: утечками ресурсов, дублирующимся кодом, риском SQL-инъекций и отсутствием централизованного управления транзакциями. В этой лекции рассматриваются как фундаментальные механизмы взаимодействия с БД через JDBC, так и способы их эффективной инкапсуляции с помощью Spring JDBC, а также инструменты миграции и декларативного управления транзакциями.

## 1. Введение в JDBC

### Проблема
Каждая СУБД (PostgreSQL, MySQL, Oracle и др.) предоставляет собственный драйвер и — исторически — собственный API для взаимодействия из приложения. Это означает, что при смене базы данных разработчику приходится переписывать значительную часть кода, связанного с подключением, выполнением запросов и обработкой результатов. Такой подход нарушает принципы модульности и переносимости.

### Определение термина: **JDBC (Java Database Connectivity)**
JDBC — это стандартный прикладной программный интерфейс (API), входящий в состав Java SE, предназначенный для унифицированного взаимодействия с реляционными базами данных. Он позволяет устанавливать соединение с БД, выполнять SQL-запросы и обрабатывать результаты, абстрагируясь от конкретной реализации СУБД.

### Преимущества JDBC
- **Независимость от СУБД**: при наличии соответствующего JDBC-драйвера достаточно изменить URL подключения и имя класса драйвера — бизнес-логика остаётся неизменной.
- **Единый API**: все операции выполняются через стандартные интерфейсы (`Connection`, `Statement`, `ResultSet`), что упрощает обучение и поддержку.
- **Упрощение разработки**: разработчику не нужно знать протоколы обмена с БД — достаточно знания SQL и базовых классов.
- **Гибкое управление транзакциями**: поддержка методов `commit()`, `rollback()`, `setAutoCommit()`.
- **Широкая поддержка**: интеграция с популярными фреймворками (Spring, Jakarta EE) и СУБД.

### Недостатки JDBC
- **Слабая абстракция**: работа происходит на уровне «сырых» SQL-запросов, без автоматического преобразования строк таблицы в объекты доменной модели.
- **Риск утечек ресурсов**: если не закрыть `Connection`, `Statement` или `ResultSet`, соединение останется открытым, что может привести к исчерпанию пула соединений.
- **Сложность масштабирования**: в больших проектах объём шаблонного кода для обработки ошибок, закрытия ресурсов и маппинга данных становится значительным, снижая читаемость и сопровождаемость.

> **Теоретическое пояснение**: JDBC реализует паттерн *Bridge* — интерфейсы (`java.sql.Connection`) отделены от реализаций (драйверы PostgreSQL, MySQL и так далее). Это позволяет менять реализацию без изменения клиентского кода.

## 2. Пример настройки соединения с БД в Java

### Проблема
Даже простейшая операция вставки записи в таблицу требует написания множества строк кода: загрузка драйвера, установка соединения, создание запроса, привязка параметров, выполнение, закрытие ресурсов. При этом легко допустить ошибку: забыть закрыть соединение, использовать конкатенацию вместо параметризации, не обработать исключение.

### Зависимости (Maven/Gradle)

```xml
<!-- Maven -->
<dependency>
    <groupId>org.postgresql</groupId>
    <artifactId>postgresql</artifactId>
    <version>42.7.9</version>
</dependency>
```

```groovy
// Gradle
implementation 'org.postgresql:postgresql:42.7.9'
```

> Эти зависимости подключают JDBC-драйвер PostgreSQL, необходимый для установки соединения.

### SQL-скрипт создания таблицы

```sql
CREATE TABLE IF NOT EXISTS users (
    id BIGSERIAL PRIMARY KEY,
    firstName VARCHAR(200) NOT NULL,
    lastName VARCHAR(200) NOT NULL
);
```

> Таблица хранит пользователей. `BIGSERIAL` — автоинкрементное поле типа `BIGINT`.

### Java-код подключения и вставки

```java
public class JdbcExampleMain {
    public static void main(String[] args) throws SQLException, ClassNotFoundException {
        // Загружает и регистрирует драйвер PostgreSQL в DriverManager
        Class.forName("org.postgresql.Driver");

        // Устанавливает соединение с БД по JDBC URL
        final Connection connection = DriverManager.getConnection(
            "jdbc:postgresql://localhost:5432/spring-java", "postgres", "postgres"
        );

        // Создаёт подготовленный SQL-запрос с параметрами (?)
        final PreparedStatement preparedStatement = connection.prepareStatement("""
            INSERT INTO users ("firstName", "lastName")
            VALUES (?, ?);
        """);

        // Подставляет значение "Иван" вместо первого ?
        preparedStatement.setString(1, "Иван");

        // Подставляет значение "Иванов" вместо второго ?
        preparedStatement.setString(2, "Иванов");

        // Выполняет запрос и возвращает количество затронутых строк
        preparedStatement.executeUpdate();
    }
}
```

#### Пояснение по каждой строке
- **`Class.forName("org.postgresql.Driver")`** — загружает класс драйвера и вызывает его статический блок, который регистрирует драйвер в `DriverManager`. В современных версиях Java (JDBC 4.0+) это не обязательно, но часто оставляют для совместимости.
- **`DriverManager.getConnection(...)`** — создаёт новое соединение с БД. Это дорогостоящая операция: сетевое рукопожатие, аутентификация, инициализация сессии.
- **`connection.prepareStatement(...)`** — создаёт объект `PreparedStatement`, который компилирует SQL-запрос один раз и позволяет многократно выполнять его с разными параметрами.
- **`setString(index, value)`** — безопасно привязывает значение к параметру. Индексация начинается с 1.
- **`executeUpdate()`** — выполняет DML-запрос (`INSERT`, `UPDATE`, `DELETE`). Возвращает число затронутых строк.

> **Теоретическое пояснение**: JDBC использует паттерн *Factory Method* — `DriverManager` выступает как фабрика, создающая `Connection` на основе URL. Также используется *Template Method* — `PreparedStatement` инкапсулирует повторяющуюся логику выполнения запроса.

## 3. Чтение данных через `Statement` и `ResultSet`

### Проблема
После вставки данных возникает необходимость их чтения. Однако даже простой `SELECT` требует аккуратной работы с курсором (`ResultSet`), управления ресурсами и корректного извлечения значений. При этом использование `Statement` вместо `PreparedStatement` делает код уязвимым к SQL-инъекциям, а отсутствие `try-with-resources` — к утечкам.

### Пример чтения

```java
try (Statement stmt = connection.createStatement()) {
    // Выполняет SELECT-запрос и получает ResultSet
    ResultSet result = stmt.executeQuery("SELECT * FROM users");

    // Проходит по всем строкам результата
    while (result.next()) {
        // Извлекает значения по имени колонки
        System.out.printf("id='%s', firstName='%s', lastName='%s'%n",
            result.getString("id"),
            result.getString("firstName"),
            result.getString("lastName"));
    }
}
```

### Анализ проблемы
- **Создание `Connection` через `DriverManager`** — дорого, не масштабируется.
- **Требуется `try-with-resources`** — иначе ресурсы не закроются.
- **`Connection` не потокобезопасен** — нельзя делить между потоками.
- **Нет автоматического восстановления** при сетевых сбоях.

> **Теоретическое пояснение**: `ResultSet` — это курсор, указывающий на текущую строку. Он не загружает все данные в память сразу (в большинстве драйверов), но требует открытого соединения. Это ограничивает использование в распределённых системах.

## 4. Пулы соединений

### Проблема
Каждый вызов `DriverManager.getConnection()` создаёт новое соединение — операцию, требующую сетевого взаимодействия и аутентификации. В многопоточном приложении с высокой нагрузкой это приводит к исчерпанию ресурсов сервера БД и увеличению задержек.

### Определение термина: **пул соединений (Connection Pool)**
Пул соединений — это механизм повторного использования уже установленных соединений с базой данных. Вместо создания нового соединения при каждом запросе приложение берёт готовое из пула, а после завершения работы возвращает его обратно.

### Обязанности пула
- **Инициализация**: создаёт пул соединений при старте.
- **Выдача**: `DataSource.getConnection()` возвращает готовое соединение.
- **Возврат**: `connection.close()` не закрывает соединение физически, а возвращает его в пул.
- **Переиспользование**: снижает задержку и нагрузку на БД.
- **Контроль ресурсов**: ограничивает максимальное число соединений, закрывает простаивающие.
- **Валидация**: перед выдачей проверяет работоспособность (`SELECT 1`).
- **Thread-safety**: каждый поток получает своё соединение.
- **Автовосстановление**: реконнект при сбоях.

### Популярные реализации
- **HikariCP** — используется по умолчанию в Spring Boot. Отличается минимальной задержкой и высокой производительностью.
- **Tomcat JDBC Pool** — часть Apache Tomcat.
- **C3P0** — старая, но стабильная библиотека.

> **Теоретическое пояснение**: пул соединений реализует паттерн *Object Pool*. Это критически важно для производительности, так как создание TCP-соединения и аутентификация в СУБД — самые дорогие операции в JDBC.

## 5. SQL-инъекции и безопасность

### Проблема
При конкатенации пользовательских данных в SQL-строку возникает угроза выполнения произвольного кода в БД. Это одна из самых опасных уязвимостей в веб-приложениях.

### Уязвимый код

```java
final String name = "admin'; DROP TABLE users;--";
final Statement stmt = connection.createStatement();
stmt.executeQuery("SELECT * FROM users WHERE \"firstName\"='" + name + "'");
```

> Этот код объединяет SQL и данные в одну строку. Если `name` содержит вредоносный SQL, он будет выполнен.

### Безопасный код

```java
final String name = "admin'; DROP TABLE users;--";
final PreparedStatement pstmt = connection.prepareStatement("SELECT * FROM users WHERE \"firstName\" = ?");
pstmt.setString(1, name);
```

> Здесь `name` передаётся как параметр, а не как часть SQL. Драйвер экранирует его автоматически.

### Определение термина: **SQL-инъекция**
SQL-инъекция — это уязвимость, позволяющая злоумышленнику внедрить произвольный SQL-код в запрос путём манипуляции входными данными.

### Сравнение `Statement` vs `PreparedStatement`

| Аспект                | `Statement`                     | `PreparedStatement`                        |
|----------------------|----------------------------------|--------------------------------------------|
| Параметры            | Нет                              | Поддержка `?` и `setXxx()`                 |
| Производительность   | Парсинг каждый раз               | Предварительная компиляция + кеш плана     |
| Безопасность         | Уязвим к инъекциям               | Защита от инъекций                         |
| Повторное использование | Нет                            | Один объект — много вызовов                |

> **Теоретическое пояснение**: `PreparedStatement` отправляет SQL и данные отдельно на уровне протокола (например, в PostgreSQL — через extended query protocol). Это исключает возможность интерпретации данных как кода.

## 6. Spring JDBC

### Проблема
Несмотря на наличие пулов и `PreparedStatement`, «голая» JDBC всё ещё требует написания большого количества шаблонного кода: открытие/закрытие ресурсов, обработка исключений, маппинг `ResultSet → Object`. Это снижает продуктивность и повышает вероятность ошибок.

### Определение термина: **Spring JDBC**
Spring JDBC — это модуль фреймворка Spring, предоставляющий высокоуровневую обёртку над JDBC для устранения boilerplate-кода и типичных ошибок.

### Проблемы «голого» JDBC
- Много шаблонного кода для открытия/закрытия ресурсов.
- Риск утечек ресурсов.
- Сложный маппинг `ResultSet → Object`.
- Отсутствие управления транзакциями.
- Зависимость от диалекта SQL.
- Сложность тестирования.

### Как Spring JDBC решает эти проблемы
- **Автоматическое управление ресурсами**: `JdbcTemplate` сам открывает и закрывает всё.
- **Единая обработка исключений**: `SQLException` → `DataAccessException` (unchecked).
- **Упрощённый маппинг**: через `RowMapper`.
- **Интеграция с транзакциями Spring.**
- **Поддержка параметризованных запросов** → защита от инъекций.
- **Лёгкая подмена `DataSource`** для тестов (например, H2).

> **Теоретическое пояснение**: `JdbcTemplate` реализует паттерн *Template Method*. Базовый алгоритм (открытие соединения, выполнение, закрытие) зафиксирован, а вариативная часть (маппинг результата) передаётся через callback (`RowMapper`).

## 7. Настройка Spring JDBC

### Проблема
Для начала работы со Spring JDBC необходимо правильно настроить зависимость и источник данных (`DataSource`), чтобы Spring мог автоматически создавать `JdbcTemplate`.

### Зависимость

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-jdbc</artifactId>
</dependency>
```

### Конфигурация (`application.yml`)

```yaml
spring:
  application:
    name: spring-demo
  datasource:
    url: jdbc:postgresql://localhost:5432/spring-java
    username: postgres
    password: postgres
    driver-class-name: org.postgresql.Driver
```

> **Best Practice**
> - Не храни пароли в коде. Используй переменные окружения, `application-prod.yml` или Vault.
> - Для dev-среды используй in-memory БД: `jdbc:h2:mem:testdb`.

> **Теоретическое пояснение**: Spring Boot автоматически создаёт бин `DataSource` на основе свойств `spring.datasource.*`. Это достигается через `DataSourceAutoConfiguration`.

## 8. Сущность и таблица

### Проблема
Для корректного маппинга между Java-объектом и строкой таблицы необходимо обеспечить соответствие имён полей и колонок. Несоответствие (camelCase vs snake_case) — частая причина ошибок.

### Java-класс

```java
@Data
@AllArgsConstructor
public class User {
    private Long id;
    private String firstName;
    private String lastName;
}
```

> Используются аннотации Lombok для генерации геттеров, сеттеров, конструктора.

### SQL-таблица

```sql
CREATE TABLE IF NOT EXISTS users (
    id BIGSERIAL PRIMARY KEY,
    firstname VARCHAR(200) NOT NULL,
    lastname VARCHAR(200) NOT NULL
);
```

> Обрати внимание: в SQL используются строчные имена колонок (`firstname`), в Java — camelCase (`firstName`). Это важно при маппинге.

> **Теоретическое пояснение**: такое несоответствие имён — частая причина ошибок. В production-коде рекомендуется использовать аннотации (`@Column(name = "firstname")`) или соглашения об именовании (snake_case в БД, camelCase в Java).

## 9. Пример запроса с `JdbcTemplate`

### Проблема
Даже с использованием `JdbcTemplate` можно написать неоптимальный или небезопасный код: выбирать все колонки, не обрабатывать `NULL`, не ограничивать выборку.

### Пример

```java
@Repository
@RequiredArgsConstructor
public class UserRepository {
    private final JdbcTemplate jdbcTemplate;

    public List<User> findAll() {
        return jdbcTemplate.query(
            "SELECT * FROM users",
            (rs, rowNum) -> new User(
                rs.getLong("id"),          // извлекает значение колонки "id" как long
                rs.getString("firstname"), // извлекает "firstname"
                rs.getString("lastname")   // извлекает "lastname"
            )
        );
    }
}
```

### Возможные проблемы
- **Нет пагинации**: при миллионе записей — `OutOfMemoryError`. Решение: `LIMIT ? OFFSET ?`.
- **`SELECT *`**: передаётся больше данных, чем нужно. Лучше указывать колонки явно.
- **Обработка `NULL`**: если колонка допускает `NULL`, `getString()` вернёт `null`. Конструктор должен это учитывать.
- **Нет логирования ошибок**: `JdbcTemplate` перехватывает исключения, но не логирует их.
- **Нет кеширования**: каждый вызов идёт в БД.

> **Теоретическое пояснение**: `rowNum` — это порядковый номер строки (начиная с 0), полезен при отладке или сложном маппинге. Однако он не связан с первичным ключом.

## 10. Методы `JdbcTemplate`

### Проблема
Разработчик должен знать, какой метод использовать для разных типов запросов: DML vs DQL.

### Вставка

```java
public void insert(User user) {
    jdbcTemplate.update(
        "INSERT INTO users(firstname, lastname) VALUES (?, ?)",
        user.getFirstName(),  // первый ?
        user.getLastName()    // второй ?
    );
}
```

> `update()` используется для DML-запросов. Возвращает число затронутых строк.

### Основные методы
- **`update(sql, args...)`** — для `INSERT`, `UPDATE`, `DELETE`.
- **`query(sql, rowMapper, args...)`** — для `SELECT`, возвращает `List<T>`.
- **`queryForObject(sql, rowMapper, args...)`** — для одного результата.

### Что делает `JdbcTemplate`
1. Получает `Connection` из `DataSource`.
2. Создаёт `PreparedStatement`.
3. Выполняет запрос.
4. Закрывает все ресурсы.
5. Преобразует `SQLException` в `DataAccessException`.

> **Теоретическое пояснение**: все методы `JdbcTemplate` синхронные и блокирующие. Для реактивного программирования следует использовать R2DBC, а не JDBC.

## 11. `RowMapper` и `ResultSetExtractor`

### Проблема
При сложных запросах (группировки, деревья, агрегаты) стандартный `RowMapper` недостаточен.

### Определение термина: **`RowMapper`**
`RowMapper` — функциональный интерфейс Spring JDBC, предназначенный для преобразования одной строки `ResultSet` в Java-объект.

### Определение термина: **`ResultSetExtractor`**
`ResultSetExtractor` — интерфейс для обработки всего `ResultSet` целиком, когда требуется контроль над курсором.

### Примеры

```java
// RowMapper
(rs, rowNum) -> new User(rs.getLong("id"), rs.getString("firstname"), rs.getString("lastname"))
```

```java
// ResultSetExtractor
jdbcTemplate.query("SELECT * FROM users", rs -> {
    List<User> result = new ArrayList<>();
    while (rs.next()) {
        result.add(new User(rs.getLong("id"), rs.getString("firstname"), rs.getString("lastname")));
    }
    return result;
});
```

> **Теоретическое пояснение**: `RowMapper` — это функциональный интерфейс, подходящий для лямбд. `ResultSetExtractor` даёт полный контроль над курсором, но требует ручного управления циклом.

## 12. `NamedParameterJdbcTemplate`

### Проблема
При большом числе параметров легко ошибиться в порядке их передачи, особенно при изменении структуры запроса.

### Пример проблемы

```java
jdbcTemplate.update("INSERT INTO t(a,b,c,d,e) VALUES (?,?,?,?,?)", a, b, c, d, e);
```

### Решение

```java
public void updateUser(User user) {
    Map<String, Object> params = new HashMap<>();
    params.put("firstName", user.getFirstName());
    params.put("lastName", user.getLastName());
    params.put("someString", user.getSomeString());

    namedJdbcTemplate.update(
        "INSERT INTO users (firstname, lastname, somestring) VALUES (:firstName, :lastName, :someString)",
        params
    );
}
```

> **Best Practice**: использовать при 4 и более параметрах. Имена должны быть осмысленными: `:startDate`, `:status`.

> **Теоретическое пояснение**: `NamedParameterJdbcTemplate` внутренне преобразует именованные параметры в позиционные (`?`), поэтому работает поверх любого `JdbcTemplate`. Это удобная обёртка, а не новый уровень абстракции.

## 13. Миграции БД: Flyway

### Проблема
Изменение структуры БД вручную (через pgAdmin, DBeaver и так далее) приводит к рассогласованию схемы между окружениями (dev/test/prod), что вызывает ошибки развёртывания и трудности отладки.

### Определение термина: **миграция базы данных**
Миграция — это версионируемый SQL-скрипт, описывающий изменение схемы БД (создание таблицы, добавление колонки и так далее), хранимый под контролем системы управления версиями.

### Зависимость

```xml
<dependency>
    <groupId>org.flywaydb</groupId>
    <artifactId>flyway-core</artifactId>
</dependency>
```

### Конфигурация

```yaml
spring:
  flyway:
    enabled: true
    locations: classpath:db/migration
```

### Пример миграции (`V1__create_users.sql`)

```sql
CREATE TABLE IF NOT EXISTS users (
    id BIGSERIAL PRIMARY KEY,
    firstname VARCHAR(200) NOT NULL,
    lastname VARCHAR(200) NOT NULL
);
```

### Алгоритм работы Flyway

1. При старте приложения Flyway читает список SQL-файлов из `classpath:db/migration`.
2. Файлы должны иметь формат: `V<номер>__<описание>.sql` (например, `V1__init.sql`).
3. Flyway подключается к БД и проверяет наличие служебной таблицы `flyway_schema_history`.
4. Если таблицы нет — создаёт её.
5. Сравнивает версии миграций в файловой системе и в таблице.
6. Применяет только те миграции, которые ещё не были выполнены.
7. После успешного выполнения каждой миграции записывает её в `flyway_schema_history`.
8. Если миграция завершается с ошибкой — откат не производится (Flyway не управляет транзакциями миграций). Поэтому каждая миграция должна быть **атомарной и обратимой**.

> **Теоретическое пояснение**: Flyway использует принцип *immutable infrastructure* для схемы БД: каждая миграция — неизменяемый шаг. Это обеспечивает воспроизводимость состояния БД на любом окружении.

> **Важно**: никогда не редактировать уже применённые миграции. Только создавать новые.

## 14. Управление транзакциями: `@Transactional`

### Проблема
Ручное управление транзакциями (`setAutoCommit(false)`, `commit()`, `rollback()`) приводит к дублированию кода, усложнению логики и высокому риску ошибок (например, забыть вызвать `rollback()`).

### Определение термина: **транзакция**
Транзакция — это последовательность операций с БД, которая рассматривается как единое целое: либо все операции выполняются успешно (commit), либо ни одна не применяется (rollback).

### Пример

```java
@Service
public class UserService {
    private final UserRepository userRepository;

    @Transactional
    public void createUsers(List<String> names) {
        for (String name : names) {
            if (name == null) {
                throw new IllegalArgumentException("User name is null");
            }
            userRepository.save(new User(name, "Иванов"));
        }
    }
}
```

### Что делает `@Transactional`
1. Spring открывает транзакцию (`setAutoCommit(false)`).
2. Выполняет метод.
3. Если метод завершается успешно — вызывает `commit()`.
4. Если выбрасывается `RuntimeException` или `Error` — вызывает `rollback()`.

> Checked-исключения **не** приводят к откату, если не указано иное через `rollbackFor`.

### Параметры `@Transactional`

#### `isolation` — уровень изоляции

| Уровень              | Описание |
|----------------------|----------|
| `DEFAULT`            | Уровень по умолчанию в СУБД (обычно `READ_COMMITTED`) |
| `READ_UNCOMMITTED`   | Разрешает «грязное чтение» (чтение незафиксированных данных) |
| `READ_COMMITTED`     | Гарантирует чтение только зафиксированных данных |
| `REPEATABLE_READ`    | Гарантирует повторяемость чтения (одна и та же строка при повторном чтении) |
| `SERIALIZABLE`       | Полная изоляция, как последовательное выполнение |

#### `propagation` — поведение при наличии текущей транзакции

| Значение           | Описание |
|--------------------|----------|
| `REQUIRED`         | Использует текущую транзакцию или создаёт новую (по умолчанию) |
| `SUPPORTS`         | Работает в транзакции, если есть; иначе без неё |
| `MANDATORY`        | Требует существующую транзакцию; иначе ошибка |
| `REQUIRES_NEW`     | Всегда создаёт новую, приостанавливая текущую |
| `NOT_SUPPORTED`    | Выполняется без транзакции, приостанавливая текущую |
| `NEVER`            | Запрещает транзакцию; иначе ошибка |
| `NESTED`           | Создаёт вложенную транзакцию (если поддерживается СУБД, например PostgreSQL) |

#### Другие параметры
- `timeout` — максимальное время выполнения транзакции (в секундах).
- `readOnly` — оптимизация для запросов только на чтение.
- `rollbackFor`/`noRollbackFor` — указание, при каких исключениях делать откат.

> **Теоретическое пояснение**: Spring использует AOP (Aspect-Oriented Programming) для реализации `@Transactional`. Прокси-объект перехватывает вызов метода и оборачивает его в транзакцию. Это работает только при вызове **извне** бина — вызов другого метода `@Transactional` внутри того же класса не сработает.

> **Рекомендация**: аннотировать `@Transactional`-**методы сервисного слоя**, а не контроллеров или репозиториев.

## 15. Заключение

Работа с базами данных в Java-приложениях прошла путь от низкоуровневого JDBC до высокоуровневых абстракций Spring. Каждый уровень решает конкретные проблемы:

- **JDBC** предоставляет единый API, но требует ручного управления ресурсами;
- **пулы соединений** решают проблему производительности при частом создании соединений;
- **Spring JDBC** устраняет boilerplate и гарантирует безопасность;
- **Flyway** обеспечивает согласованность схемы БД между окружениями;
- **`@Transactional`** предоставляет декларативное управление транзакциями.

Эти инструменты вместе образуют надёжный, масштабируемый и поддерживаемый подход к работе с реляционными базами данных в Java-приложениях на основе Spring.
