> Apache Kafka. Понятие: topic, partition, offset, consumer group. Понятие Replication и In-Sync-Replicas (ISR). Отличие от других инструментов.

## Что такое Kafka

**Apache Kafka** — распределённая платформа потоковой передачи событий (брокер сообщений +
лог событий). Основа **событийно-ориентированной архитектуры (EDA)**: продьюсеры пишут
события, консьюмеры их читают, компоненты слабо связаны.

## Topic (топик)

**Топик** — именованная категория/лента сообщений (например, `user-events`). Продьюсеры
пишут в топик, консьюмеры читают. Это лог, доступный только для **дозаписи (append-only)**.

## Partition (партиция)

Топик делится на **партиции** — единицы параллелизма и масштабирования. Каждая партиция —
упорядоченная последовательность сообщений на конкретном брокере. Сообщения с одинаковым
**ключом** попадают в одну партицию (гарантия порядка в пределах ключа). Порядок
гарантируется **только внутри партиции**, не по всему топику.

## Offset (смещение)

**Offset** — порядковый номер сообщения **внутри партиции** (монотонно растёт). Консьюмер
хранит свой offset (коммитит его), чтобы знать, докуда прочитал, и продолжить после
перезапуска. Kafka **не удаляет** сообщение после чтения (в отличие от очередей) — данные
живут по retention-политике (время/размер), их могут читать многие консьюмеры независимо.

## Consumer group (группа консьюмеров)

**Группа** — набор консьюмеров с общим `group.id`, совместно читающих топик. Каждая
**партиция назначается ровно одному** консьюмеру в группе → масштабирование чтения.
- Консьюмеров в группе ≤ числа партиций (лишние простаивают).
- Разные группы читают **независимо** (каждая со своими offset'ами) — паттерн
  publish/subscribe.
- При изменении состава группы происходит **rebalance** (перераспределение партиций).

## Replication и ISR

**Replication** — каждая партиция реплицируется на несколько брокеров (`replication.factor`)
для отказоустойчивости. У партиции есть **leader** (принимает запись/чтение) и **followers**
(копируют данные).

**ISR (In-Sync Replicas)** — множество реплик, которые **достаточно догнали** лидера
(синхронны). Запись считается зафиксированной по правилу `acks`:
- `acks=0` — без подтверждения (быстро, риск потери);
- `acks=1` — подтверждение от лидера;
- `acks=all` — подтверждение от **всех ISR** (надёжно, медленнее).
`min.insync.replicas` задаёт минимальный размер ISR для приёма записи. Если лидер падает,
новым лидером становится одна из ISR — без потери подтверждённых данных.

## Отличие от других инструментов

| | Kafka | RabbitMQ / классич. очереди |
|---|---|---|
| Модель | распределённый **лог** (данные хранятся) | брокер очередей (сообщение удаляется после ack) |
| Чтение | многократное, разными группами, по offset | обычно одно потребление |
| Масштабирование | партиции + consumer groups | очереди/exchange |
| Назначение | потоковая обработка, event sourcing, аналитика | задачи/RPC, гибкая маршрутизация |
| Порядок | в пределах партиции | в пределах очереди |
| Throughput | очень высокий | средний |

Kafka хранит события и позволяет «переигрывать» историю; RabbitMQ — про доставку и сложную
маршрутизацию сообщений.
